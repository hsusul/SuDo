# SuDo Design Skill Notes

This document records the external design guidance used for SuDo frontend work. The linked resources are treated as design and process references, not as executable dependencies. Do not run unknown scripts from external repositories during product work.

## Resources Reviewed

- Impeccable: https://impeccable.style/#why
- Taste Skill: https://github.com/leonxlnx/taste-skill
- Emil Kowalski UI Skill: https://github.com/emilkowalski/skill
- Local frontend-design skills available in Codex for this workspace.

No external skill files were installed into the repository for this pass. The public resources provide enough process guidance, and keeping the repo free of third-party instruction bundles avoids unnecessary churn.

## Extracted Principles

### Impeccable

- Avoid generic AI frontend output: default gradients, vague dashboards, fake metrics, random glow effects, and ungrounded marketing copy.
- Respect the existing product and design system before changing components.
- Improve concrete UI dimensions: hierarchy, color, typography, spacing, motion, and copy.
- Prefer visible product clarity over decorative polish.

### Taste Skill

- Push beyond generic framework defaults by making deliberate visual choices.
- Use spacing, density, and hierarchy to clarify workflow.
- Keep interfaces specific to the product domain instead of applying a universal SaaS template.
- Audit the design before editing so the work responds to real flaws.

### Emil Kowalski UI Skill

- Treat microinteractions, focus states, and details as part of product quality.
- Keep information design clean: predictable navigation, clear primary actions, and compact controls.
- Make UI states feel refined without overanimating or hiding functionality.
- Use restraint: a polished product surface should not need ornamental excess.

## SuDo Application Rules

- Use one visual vocabulary: quiet charcoal surfaces, warm off-white text, low-contrast borders, and restrained sage/amber/rose/blue semantic accents.
- Preserve the minimal flower mark as a brand cue, not a decorative pattern.
- Keep issue tracking dense enough for repeated use. Lists and detail panels matter more than hero decoration.
- Prefer plus-icon creation entry points and focused dialogs over persistent full-width forms.
- Keep URL-backed project, issue, and filter state intact.
- Avoid copying Linear directly. Use it as a quality bar for density and clarity, not a component-by-component source.
- Keep accessibility visible: focus rings, contrast, keyboard row access, and clear labels.
- Do not introduce design dependencies unless a task clearly justifies them.
